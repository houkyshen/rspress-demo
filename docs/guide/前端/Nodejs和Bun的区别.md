# Bun vs Node.js 核心区别（精简实用版）
一句话总结：**Node.js是成熟稳定的老牌JS运行时；Bun是Zig+JSC全新自研、一体化全能工具链，主打极致速度、内置全套工程工具，兼容绝大多数Node生态，但上线生产稳定性弱于Node**。

## 一、底层内核差异（本质根源）
| 项目         | Node.js                                                                  | Bun                                                                               |
| ------------ | ------------------------------------------------------------------------ | --------------------------------------------------------------------------------- |
| JS引擎       | Google **V8**（Chrome内核，侧重长时间运行峰值性能、JIT预热慢、冷启动差） | Apple **JavaScriptCore(JSC)**（Safari内核，移动端优化，**冷启动飞快、内存更低**） |
| 内核开发语言 | C++                                                                      | **Zig**（手动精细内存管理、零拷贝IO，底层优化拉满）                               |
| 事件循环     | libuv(C语言跨平台异步库)                                                 | 自研Zig异步运行时，无libuv依赖                                                    |

## 二、性能差距（最直观区别）
1. **启动速度**：Bun冷启动是Node的**4～10倍**，简单http服务Bun约6ms、Node近90ms；脚本/CLI工具优势巨大。
2. **HTTP吞吐**：裸服务Bun QPS≈5w+，Node≈1.4w；搭配中间件Bun依旧是Node 2倍以上。
3. **依赖安装**：`bun install`速度是npm **20～40倍**，远快pnpm，node_modules体积更小。
4. **内存占用**：同等业务Bun内存普遍少30%～50%。
5. **测试速度**：内置test比Jest快5～8倍。

## 三、工具链设计（最大使用体验区别）
### Node.js：**拆分式工具链（运行时单独，工具全靠第三方）**
- 仅提供`node`运行命令
- 装依赖：npm/yarn/pnpm（独立程序）
- TS运行：需ts-node/tsx额外安装配置
- 打包：webpack/vite/esbuild（单独工具）
- 单元测试：Jest/Vitest（单独安装）

### Bun：**All in One一体化单文件工具**（一条`bun`命令搞定全流程）
- `bun run`：替代node，**原生直接跑.ts/.tsx/jsx，零配置TS**
- `bun install`：内置高速包管理器，兼容npm源、package.json
- `bun build`：内置打包器，替代vite/webpack
- `bun test`：内置测试框架，兼容Jest语法
- 额外内置：`bun:sqlite`原生数据库、WebSocket高性能服务、文件API优化

## 四、语法&兼容性
1. **Node**：CommonJS为主，ESM需配置开启，TS不原生支持
2. **Bun**：**CommonJS+ESM双原生自动识别，TS/JSX开箱即用**，不用tsconfig编译即可执行；兼容约95% Node原生API、97%+ npm包，老Node项目大多直接替换运行。
> 小坑：少量C++原生插件（node-addon）Bun无法兼容，这类库只能继续用Node。

## 五、生态&成熟度（生产选型关键）
- **Node.js（2009诞生，15+年）**：生态海量npm包、企业级LTS、云厂商全适配、故障方案完善，生产首选，无兼容踩坑风险。
- **Bun（2023正式1.0）**：发展快但年限短，小众边缘包、老C++插件偶发兼容bug；**开发环境、脚本、新项目优先Bun，老旧核心业务稳妥Node**。

## 六、适用场景选型
### ✅ 优先用Bun
1. 新项目、前端全栈、Hono/Elysia轻量服务
2. 本地开发、脚本自动化、CI打包、单元测试提速
3. CLI工具、高频启停的短生命周期服务

### ✅ 优先用Node.js
1. 线上核心生产、老旧大型Node项目迁移成本高
2. 依赖大量C++二进制插件（图像处理、数据库驱动等）
3. 金融/政企等对稳定性、长期兼容要求极高业务

## 补充小常识
现有Node项目不用全量迁移：**本地用bun install/bun test提速，生产仍用node跑**，混合使用是目前业界主流方案。
