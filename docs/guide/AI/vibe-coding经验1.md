spec: 如下
# AI 文件处理 Agent 项目规格说明书（SPEC）

## 1. 项目概述

### 1.1 项目名称

自然语言驱动 AI 文件处理智能体（AI File Agent）

### 1.2 项目定位

基于 Python + Pydantic AI 框架开发的轻量级 AI Agent，无需用户操作鼠标、编写代码，仅通过**自然语言口语/文字指令**，即可自动完成本地文件夹的文件读取、列表查看、重命名、信息解析等批量文件管理操作。

项目核心价值：**将自然语言自动翻译为文件系统操作，实现零代码、智能化本地文件管理**，同时具备 AI 上下文记忆能力，支持多轮连续文件操作。

### 1.3 适用场景

- 批量整理杂乱的代码文件、文档文件、素材文件

- 自然语言查询文件类型、文件内容、文件功能

- 根据文件内容智能批量重命名、归类文件

- 多轮连续文件操作，AI 自动记忆历史操作与对话上下文

## 2. 技术栈规范

- **开发语言**：Python 3.10+

- **Agent 框架**：Pydantic AI

- **大模型服务**：阿里通义千问 DashScope API（兼容 Pydantic AI 标准接口，默认使用 qwen-plus 模型）

- **依赖与环境管理**：uv 工具管理依赖 + 独立虚拟环境，统一版本、轻量化部署

- **核心能力支撑**：大模型函数调用（Function Calling）

## 3. 核心功能需求（Functional Requirements）

本 AI Agent 仅聚焦**本地文件安全操作**，所有功能基于自定义工具函数实现，由大模型自主判断调用时机。

### 3.1 基础文件工具能力（内置三大核心函数）

#### 3.1.1 listfiles 文件列表查询

- 功能描述：读取指定文件夹下所有文件名称，列出完整文件清单

- 入参：目标文件夹路径（默认 Test 测试目录）

- 返回值：文件夹内所有文件名列表

- 应用场景：用户询问「查看这个文件夹有哪些文件」

#### 3.1.2 readfile 文件内容读取

- 功能描述：读取指定单个文件的完整内容

- 入参：文件路径、文件名

- 返回值：文件文本内容，文件不存在则返回标准错误提示

- 应用场景：用户询问「这个文件是什么代码」「B 文件的功能是什么」

#### 3.1.3 renamefile 文件重命名

- 功能描述：对指定文件进行重命名操作

- 入参：原文件路径、原文件名、新文件名

- 返回值：重命名成功/失败提示信息

- 应用场景：用户指令「根据内容给所有文件加上对应后缀」「批量重命名文件」

### 3.2 AI 智能决策能力

- 自动识别用户自然语言意图，匹配对应文件工具函数

- 支持多步骤链式操作：先列文件、再读内容、最后批量重命名

- 自主判断是否需要重复调用工具，无需人工干预

### 3.3 上下文记忆能力（核心进阶功能）

- 默认关闭独立会话，支持手动保存每轮对话历史记录

- 新一轮指令自动加载`message_history`，AI 可记忆过往文件读取、操作记录

- 避免重复读取、重复查询，提升文件操作效率

### 3.4 人机交互能力

- 控制台标准输入接收用户自然语言指令

- 控制台标准化打印 AI 思考过程、工具调用记录、最终执行结果

- 支持任意自然语言口语化指令，无需固定命令格式

## 4. 非功能需求（Non-Functional Requirements）

### 4.1 安全性

- 通义千问 API 密钥统一存放于 `.env` 文件（字段为 DASHSCOPE_API_KEY），禁止硬编码、禁止提交 Git

- 限定 Agent 仅操作指定 Test 目录，防止误删、误改系统文件

- 文件操作函数自带异常捕获，报错不崩溃、不泄露路径信息

### 4.2 可扩展性

- 工具函数模块化设计，可随时新增删除文件、移动文件、复制文件等功能

- 模型接口可一键切换，兼容通义千问全系模型、GPT、Claude 等主流大模型

- 注释标准化，支持后续功能迭代、二次开发

### 4.3 易用性

- 零前端界面依赖，纯控制台运行，低配置电脑可直接使用

- 用户无需代码基础，口语化指令即可完成专业文件操作

## 5. 项目环境与依赖管理规范（UV）

### 5.1 环境方案

本项目全程采用 **uv** 进行 Python 环境与依赖管理，摒弃原生 pip 管理方式，通过独立虚拟环境实现项目依赖隔离，避免全局包冲突、版本混乱问题，保证项目在不同设备、不同系统下可复现运行。

### 5.2 核心依赖清单（由 uv 统一管理）

- **pydantic-ai**：AI Agent 核心框架，支撑模型调用、工具注册、Function Calling

- **dashscope**：通义千问官方 SDK，适配大模型接口调用

- **python-dotenv**：环境变量读取、密钥配置隔离

### 5.3 标准化操作流程

1. 项目初始化：执行 `uv init` 生成项目配置文件与基础结构

2. 创建虚拟环境：执行 `uv venv` 生成独立 .venv 虚拟环境目录

3. 安装依赖：执行`uv add 包名` 自动写入 pyproject.toml 并安装依赖

4. 运行项目：统一使用 `uv run python main.py` 启动程序，绑定虚拟环境依赖

## 6. 系统运行流程（核心逻辑）

## 5. 系统运行流程（核心逻辑）

**完整运行链路：**

1. 通过 uv 激活本地虚拟环境，加载 .env 环境变量，读取通义千问 API 密钥

2. 初始化通义千问模型实例，配置 System Prompt（定义资深程序员角色、文件操作规则）

3. 注册三大文件工具函数到 Agent 实例，绑定工具描述、参数、注释文档

4. 接收用户自然语言输入指令

5. Agent 结合上下文历史，自主决策需要调用的工具函数

6. 自动执行文件操作，获取工具返回结果

7. 大模型整合结果，生成自然语言回复，输出给用户

8. 保存本轮对话记录，用于下一轮上下文记忆

## 7. 代码开发规范

## 6. 代码开发规范

### 6.1 函数开发规范

- 所有工具函数必须标注**明确参数类型、返回值类型**

- 函数名、参数名语义清晰、见名知意

- 复杂函数必须编写完整 Docstring 注释，供大模型读取识别

- 所有文件操作增加 try-except 异常捕获，容错运行

### 6.2 Agent 配置规范

- System Prompt 固定角色：经验丰富的程序员，专注智能文件管理

- 强制开启工具调用能力，禁止大模型凭空回答

- 通义千问模型原生支持 Function Calling，稳定适配上下文记忆，默认开启记忆优化

## 7. 测试用例规范

3. **上下文记忆测试**：多轮对话复用历史文件信息，杜绝重复读取

## 10. 工程化约束规范

### 10.1 环境约束

- 禁止使用全局 Python 环境运行项目，必须基于 uv 创建的 .venv 虚拟环境执行

- 所有依赖变更必须通过 `uv add`/`uv remove` 操作，手动 pip 安装无效

- `.venv` 虚拟环境目录默认忽略，不提交代码仓库

### 10.2 版本一致性约束

- 所有开发、运行、测试环境统一使用 uv 管理，保证团队、跨设备依赖版本完全一致

- pyproject.toml 为唯一依赖配置文件，所有依赖版本以该文件为准
